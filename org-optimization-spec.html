一、设计目标与核心原则

设计目标

保留 HiveWeave 的灵活组织架构（按项目需求搭建，非常驻固定角色），注入 gstack 的工程约束框架，使得 HR 招聘的每个角色都自带 gstack 级别的工程纪律。不是照搬 gstack 的角色列表，而是提取其"管理和约束"的精华，提升团队工作效率和能力。

核心原则

原则含义来源
灵活组织 + 固定约束组织结构按项目需求灵活搭建，但每个角色的工程纪律是固定的、不可协商的用户决策
约束即提示词工程约束通过系统提示词注入，不依赖额外的代码逻辑（除非涉及权限）gstack 设计
验证不可协商每个角色的每个动作必须有证据支撑，"看起来对"永远不够agent-skills
反合理化内置把"代理偷懒跳步的借口"显式写进提示词，配反驳——从源头阻止偷懒agent-skills 独特设计
用户参与度可调用户主权不是固定铁律，而是可配置的参与度级别。用户按自身能力决定让渡多少决策权给 AI（高/中/低三级），AI 据此调整自主决策范围用户反馈优化

二、约束框架提取（gstack/agent-skills → HiveWeave）

2.1 ETHOS 三原则 — 注入所有角色的共享文化层

来源：gstack ETHOS.md，注入到每个角色提示词的共享前言中。

原则 1: Boil the Lake（做完整的事）

AI 让"完整性"的边际成本趋近于零。当完整实现只比捷径多花几分钟时，就做完整版。区分"湖"（可煮沸：100% 测试覆盖、完整边界处理）与"海洋"（不可煮沸：整体重写、跨季度迁移）。

反模式："省 70 行代码只做 90%"、"测试留到下个 PR"、"边界情况以后再说"。

原则 2: Search Before Building（先搜索后构建）

三层知识观：Layer 1 验证过的成熟模式（直接用）；Layer 2 新流行的实践（审视后用，人群会狂热）；Layer 3 第一性原理推导（最有价值，"11/10 的项目"往往来自这种 zig while others zag）。

原则 3: User Involvement（用户参与度，可调）

用户主权不是固定铁律，而是可配置的参与度级别。用户按自身能力决定让渡多少决策权给 AI。不懂技术的用户可以把技术决策权让渡给 AI，但保留产品/使用角度的决策权。

级别技术决策产品/业务决策重大方向变更适用场景
高（默认）必须问用户必须问用户必须问用户用户有技术能力且想掌控方向
中AI 自主执行必须问用户必须问用户用户懂产品不懂技术，让渡技术决策权
低AI 自主执行AI 自主执行仅通知用户用户完全信任 AI 或只想看结果

配置方式：通过 charter 的 user_involvement 字段配置（"high" / "medium" / "low"），CEO 提示词的 build_context_prompt 读取该配置，动态调整 CEO 的决策行为——高参与度时方向决策必问；中参与度时技术自主+产品必问；低参与度时仅重大变更通知。

不变的部分：无论哪个级别，AI 都不能伪造结果、不能隐藏风险、不能跳过验证。让渡的是决策权，不是诚实义务。

2.2 角色纪律四件套 — 每个角色必备的结构

来源：agent-skills SKILL.md 解剖 + gstack 角色定义。每个角色提示词必须包含以下四部分：

组件作用示例
何时不做
(When NOT to use)		 明确角色边界，防止越权 code_reviewer: "不写代码，不提供修复，只描述问题"
输出格式
(Output format)		 结构化输出，便于下游消费 security_auditor: "Verdict + CWE 编号 + CVSS 估算 + 具体修复建议"
验证清单
(Verification checklist)		 退出标准，每项须有证据 "测试通过(附输出)"、"构建成功(附日志)"、"运行时验证(附截图)"
反合理化表
(Anti-rationalization)		 显式列出偷懒借口 + 反驳 "我稍后加测试" → "测试是代码的一部分，没有测试的代码是未完成的代码"

2.3 铁律约束 — 角色专属硬规则

来源：gstack 各 SKILL.md 的铁律。这些是不可违反的硬约束，违反即失败。

角色铁律来源
Debugger / 调试者No fixes without investigation（不调查就不修复）。3 次修复失败后停止并升级。gstack /investigate
Security Auditor8/10 置信度门槛；17 项误报排除；每条发现必须附 exploit 场景；Critical 立即升级。gstack /cso
Code Reviewer同一任务拒绝 3 次则升级；变更规模超 ~100 行建议拆分；严重级别标签强制。agent-skills + gstack /review
QA / Test EngineerBeyoncé Rule（如果你喜欢它，你就该测试它）；连续 3 次失败则升级；每个 pass/fail 必须有测试输出佐证。agent-skills + gstack /qa
Release Engineer没有测试框架时先引导搭建；覆盖率不达标不发布。gstack /ship
Web Perf Auditor指标诚实规则：绝不伪造指标；无工具数据时只返回源码级发现，标"not measured"。agent-skills web-performance-auditor

2.4 权限白名单模式

来源：gstack allowed-tools frontmatter 字段。HiveWeave 当前只有 coordinator/executor 二分，过粗。借鉴 gstack 的每角色独立白名单思路，但保留 HiveWeave 的 permission_type 作为粗粒度基础，新增 role 级别的工具细粒度控制。

2.5 验证文化

来源：agent-skills "验证不可协商"。每个技能/角色以证据要求结尾。HiveWeave 现有的 ACTION DISCIPLINE 需升级为可验证的退出标准，而非笼统的"说到必须做到"。

三、注入机制设计（三层提示词架构）

3.1 三层提示词架构

将 HiveWeave 现有的两层提示词（identity + context）扩展为三层约束注入：

┌─────────────────────────────────────────────────────────┐
│ Layer 0: ETHOS 共享层（所有角色，注入 build_identity_prompt 共享部分）  │
│   - Boil the Lake / Search Before Building / User Involvement(可调)  │
│   - 用户参与度：charter.user_involvement 决定 AI 自主决策范围              │
│   - 通用验证文化：每个动作须有证据                                       │
│   - 通用反合理化：列 3-5 条跨角色的偷懒借口 + 反驳                       │
├─────────────────────────────────────────────────────────┤
│ Layer 1: 角色类型约束层（coordinator / executor / reviewer 分支）       │
│   - coordinator: 管理纪律（不写代码、审批门禁、管理跨度）                  │
│   - executor: 执行纪律（先调查后修复、测试先行、完整实现）                 │
│   - reviewer: 审查纪律（只读不改或只改不改架构、输出格式、升级阈值）        │
├─────────────────────────────────────────────────────────┤
│ Layer 2: 角色专属剧本层（具体 role: ceo / hr / qa / test_engineer...）  │
│   - 角色铁律（不可违反的硬约束）                                          │
│   - 角色反合理化表（该角色专属的偷懒借口 + 反驳）                          │
│   - 角色验证清单（退出标准，须有证据）                                     │
│   - 角色输出格式（结构化输出模板）                                        │
└─────────────────────────────────────────────────────────┘

3.3 招聘时自动应用约束

HR 通过 hire_agent 招聘时，根据 role 关键词匹配约束模板。约束模板存储在代码中（streamer.ex 的角色分支），HR 只需传入正确的 role 和 skills，系统自动注入对应约束。

同时增强 HR 提示词中的技能绑定标准表，确保招聘时自动绑定正确的技能集。

四、7 个角色改造方案

4.1 CEO（首席执行官）

增强

现状问题

  • 开发生命周期已有（EXPLORE→DEFINE→PLAN→BUILD→VERIFY→REVIEW→SHIP），但缺乏"Boil the Lake"完整性约束
  • 组织设计规则已有，但缺乏"Search Before Building"的前置搜索要求
  • User Sovereignty 原则未显式声明，CEO 可能替用户做方向决策

注入的 gstack 约束

  • ETHOS 注入：在 CEO 提示词开头注入三原则，特别强化 User Sovereignty（"CEO 的组织设计是建议，用户确认后才执行"）
  • Boil the Lake：开发生命周期每阶段增加完整性检查点——DEFINE 阶段必须产出完整 spec（非粗略想法）；BUILD 阶段必须含边界处理；VERIFY 阶段必须有测试输出
  • Search Before Building：Phase 0 探索阶段增加"先搜索成熟模式"要求——在 design org 之前，先搜索该项目类型的常见组织模式

反合理化表（新增）

"先招人，角色定义以后再说"角色定义是招聘的前提。模糊的角色定义导致重复招聘或职责真空。先写 charter 再招人。
"这个方向很明显，不用问用户"根据用户参与度配置决定：高参与度=必须用 question 确认；中参与度=技术方向自主，产品方向必问；低参与度=自主执行但通知用户。注意：让渡决策权不等于让渡诚实义务，风险必须如实告知。
"spec 太细浪费时间，先写代码"Boil the Lake：spec 是代码的前提。省 spec 的 10 分钟会在 debug 阶段花 2 小时。

验证清单（新增）

  • 组织设计完成 → charter 已保存（read_charter 可读回）
  • 招聘指令发出 → send_message 有 HR 回执
  • 任务派发 → 每个 executor 收到 expectReport=true 的消息

4.2 HR（人力资源）

增强

现状问题

  • 技能绑定标准表已有，但缺乏"招聘质量门"——招进来的角色是否真的具备所需技能
  • 花名/岗位命名规则已有，但 backstory 质量参差

注入的 gstack 约束

  • 招聘质量门：每次 hire_agent 后，HR 必须验证新 agent 的 role/skills/goal/backstory 是否完整且匹配需求，不匹配则 dismiss 重招
  • Search Before Building：招聘前先检查现有组织是否已有同 role 的 agent（避免重复招聘）

反合理化表（新增）

"先招了再说，技能不设也行"招聘时必须设定初始技能集——这是角色定义的前提。没有初始技能的 agent 无法胜任职责。
"技能设定后就不能改了"技能不是锁死的。随着项目推进发现新问题，Agent 应具备自主添加技能的能力（通过 bind_skill 工具）。初始技能是起点，不是终点。
"backstory 随便写两句就行"backstory 让 agent 有真实人物感，影响 LLM 的角色一致性。必须 2-4 句有情节的叙事。

4.3 QA Engineer（质量保证工程师）

权限修复 约束增强

现状问题（Bug）

qa_engineer role 不在 reviewer 匹配列表（reviewer/inspector/审查员/qa）中，导致自动创建的 QA Engineer 实际走通用 executor_tools，拿不到 qa_review_tools（run_code_review/run_security_audit 等）。

修复

  • tool_executor.ex 的 reviewer 匹配列表加入 qa_engineer 和 qa（已有 qa 但 qa_engineer 缺失）
  • 同时修正 projects_controller.ex 中 QA Engineer 的 role 为 qa_engineer 并确保匹配

注入的 gstack/agent-skills 约束

  • Beyoncé Rule：如果你喜欢它，你就该测试它——关键路径必须有测试覆盖
  • 测试金字塔：单元/集成/E2E = 80/15/5，避免倒金字塔
  • DAMP over DRY：测试中描述性（DAMP）优先于不重复（DRY）
  • Prove-It 模式：为 bug 先写会失败的测试，再修代码

反合理化表（新增）

"这个改动太小，不用测试"Beyoncé Rule：小改动也能引入大 bug。每个改动都需要测试。
"手动测过了，应该没问题"手动测试不可重复。必须有自动化测试输出作为证据。
"测试覆盖率 80% 够了"覆盖率不等于质量。关键路径必须 100%，边角可以低。

4.4 test_engineer（测试工程师）

Elixir 新增 约束增强

现状问题

Elixir 端缺失完整提示词，需在 streamer.ex reviewer 分支中新建并增强。

注入的约束

  • 铁律：不写应用代码，只测试和报告；连续 3 次失败则升级
  • 输出格式：Summary / Failures / Regressions / Recommendation，每个 pass/fail 附测试输出
  • 验证清单：测试命令已执行(附输出) / 覆盖率已分析(附数据) / 回归已检查(附对比)

反合理化表

"测试框架没配好，我先跳过"没有测试框架时先引导搭建（借鉴 gstack /ship 的做法），不跳过。
"这个测试偶尔失败，先注释掉"flaky test 是信号不是噪音。调查根因，不注释。

4.5 code_reviewer（代码审查员）

Elixir 新增 约束增强

注入的约束（对标 agent-skills code-reviewer persona + gstack /review）

  • 五轴评审：正确性 / 可读性 / 架构 / 安全 / 性能（已有，强化）
  • 变更规模约束：变更超 ~100 行建议拆分（agent-skills code-review-and-quality）
  • 铁律：不写代码，不提供修复，只描述问题；同一任务拒绝 3 次则升级
  • 输出格式：Verdict (APPROVE/CHANGES REQUESTED/REJECT) + Critical/Warning/Nitpick 分级
  • 评审标准："一位 staff 工程师会批准这个吗？"（agent-skills）

反合理化表

"代码能跑就 APPROVE 吧"能跑 ≠ 正确。审查五轴，不是审查"能不能跑"。
"改动太大，我随便看看就过了"大改动更需要仔细审查。变更超 100 行建议拆分后再审。
"这个问题太小不用提"Nitpick 也要提。审查的目的是让代码更好，不是走流程。

4.6 security_auditor（安全审计员）

Elixir 新增 约束增强

注入的约束（对标 gstack /cso + agent-skills security-auditor）

  • 铁律：8/10 置信度门槛（低于不报）；17 项误报排除；每条发现必须附 exploit 场景；Critical 立即升级
  • 评审范围：OWASP Top 10 + STRIDE 威胁建模 + 密钥检测 + 依赖供应链 + LLM/AI 安全（OWASP LLM Top 10）
  • 输出格式：Verdict (CLEAR/ISSUES FOUND/CRITICAL VULNERABILITY) + CWE 编号 + CVSS 估算 + 具体修复建议
  • 聚焦可利用漏洞，而非理论风险（agent-skills）

反合理化表

"这个漏洞理论上有风险但不太可能被利用"聚焦可利用漏洞，但如果能构造 exploit 场景就必须报。不能利用的不报（误报排除）。
"Critical 发现先观察一下再说"Critical 立即升级。不等观察，不攒报告。

4.7 web_perf_auditor（Web 性能审计员）

Elixir 新增 约束增强

注入的约束（对标 agent-skills web-performance-auditor + gstack /benchmark）

  • 铁律（指标诚实）：绝不伪造指标；LLM 读静态源码无法测量真实 LCP/INP/CLS；无工具数据时只返回源码级发现，标"not measured"
  • 两种模式：Quick mode（默认，扫源码找反模式）/ Deep mode（解析 Lighthouse/CrUX/DevTools 数据）
  • Core Web Vitals 目标：LCP < 2.5s / INP < 200ms / CLS < 0.1
  • 输出格式：Verdict (PASS/NEEDS OPTIMIZATION/BLOCKING) + Core Web Vitals 表格 + 瓶颈分析

反合理化表

"这个页面应该挺快的"指标诚实：不猜。无测量数据时标"not measured"，只报源码级反模式。
"LCP 大概 2 秒左右"绝不编造数字。要么用工具测量，要么标"not measured"。

5.3 权限评估顺序

保持现有 deny → ask → allow 顺序，新增 role 级白名单在 permission_type 之后评估：

get_tools(agent) 流程:
  1. base = permission_type 决定基础工具集（coordinator_tools / executor_tools）
  2. role_match = role 关键词匹配决定角色专属工具（qa_review_tools / worktree_tools 等）
  3. whitelist = agent.allowed_tools 字段（如有，进一步收窄）
  4. blacklist = agent.denied_tools 字段（如有，移除）
  5. 最终工具集 = (base ∪ role_match) ∩ whitelist - blacklist

六、组织范式库增强

文件: apps/hiveweave/lib/hiveweave/llm/streamer.ex（CEO 提示词中的范式库）

6 种现有范式各注入"必经流程节点"，确保无论选哪种范式，工程纪律都不缺失：

范式必经流程节点（新增）
solo（单兵） DEFINE → BUILD → VERIFY → REVIEW（自审）→ SHIP。单兵也必须自审，不能跳过 REVIEW。
flat_squad（扁平小组） DEFINE（共商）→ BUILD（并行）→ REVIEW（交叉审）→ SHIP。交叉审查：A 写 B 审，B 写 A 审。
tech_lead（Tech Lead 制） PLAN（Lead 规划）→ BUILD → VERIFY → REVIEW（Lead 审）→ SHIP。Lead 必须审查每个 PR。
pm_architect（PM+架构师） DEFINE（PM）→ DESIGN（架构师）→ BUILD → VERIFY → REVIEW（架构师）→ SHIP（PM）。架构师做技术门禁，PM 做范围门禁。
pod（Pod/小组制） 每个 Pod 内部走 flat_squad 流程；Pod 间走 PLAN → INTEGRATE → REVIEW → SHIP。集成阶段必须交叉审查。
pipeline（流水线） DEFINE → BUILD → VERIFY → REVIEW → SHIP，每阶段有明确入口/出口标准，上一阶段未通过不进入下一阶段。
