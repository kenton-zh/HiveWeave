## 根因诊断(用户修复未完整)

用户上一轮声称"已修复 BUG-034", 后端 `chat_message.py:255-260` 加了 camelCase 别名(`isBackground`/`isStreaming`/`isRead`/`isContext`/`teamFromAgentId`/`teamToAgentId`)。Python 验证脚本(`verify_bug034.py`,已清理)显示 API 确实返回 55 条 team 消息, 字段都正确。

但前端浏览器仍看不到折叠区。我深入追查发现**真正的根因在 ChatPanel.tsx:108-114**:

```ts
function isTeamChannelMessage(msg: ChatMessage): boolean {
  if (msg.isContext) return false;   // ← 致命!
  return (
    msg.role === "team" ||
    (msg.isBackground && (msg.role === "user" || msg.role === "assistant"))
  ) as boolean;
}
```

后端 `trigger.py:218-225` 在保存 team 消息时**同时**设了 `is_background=1` **和** `is_context=1`(用于 inbox 注入 LLM 上下文)。前端 `isTeamChannelMessage` 的第 109 行 `if (msg.isContext) return false;` 把**所有 team 消息排除掉** → `teamMessages.length === 0` → `hasTeamComms === false` → 折叠区永远不渲染。

**用户只修了后端(字段名), 没有改前端(isContext 误判逻辑)**。这是 data semantics 冲突: 同一条消息既是 "inbox context"(用于 LLM 注入), 又是 "team channel"(用于 UI 折叠区), 两个标志不应互斥排除。

## 实际修复

**前端 ChatPanel.tsx:108-117** 删除了 `if (msg.isContext) return false;` 行, 并把 `msg.isBackground` 改为 `msg.isBackground === true` 严格布尔比较:

```ts
function isTeamChannelMessage(msg: ChatMessage): boolean {
  // BUG-034 root-cause fix: 触发器保存的 team 消息同时标记了 is_context=True
  // (用于 inbox 注入 LLM 上下文),但前端的 isTeamChannelMessage 不能因此排除
  // 它们 — 否则 teamMessages 永远为空,hasTeamComms=false,折叠区永不显示。
  // 判定完全交给 isBackground+role 双条件,已足够精确。
  return (
    msg.role === "team" ||
    (msg.isBackground === true && (msg.role === "user" || msg.role === "assistant"))
  ) as boolean;
}
```

Vite HMR 自动应用, 浏览器刷新后立刻看到折叠区。
